# Migrating a downstream project from v0.1.0 → v1.0.0

> Applies to any project bootstrapped from the v0.1.0 initial Projectsmith
> release (or any pre-v1.0.0 build). If your repo's top level contains a
> single directory whose name matches your `project_slug`, you are
> affected.

---

## Why

Earlier Projectsmith releases shipped the Copier template root as
`template/{{project_slug}}/`. Combined with `_subdirectory: "template"`
in `copier.yaml`, every `copier copy` materialized scaffold files inside
an *extra* `<project_slug>/` subdirectory of the destination, producing a
double-nested layout:

```
my_project/
└── my_project/
    ├── AGENTS.md
    ├── CLAUDE.md
    ├── pyproject.toml
    └── ...
```

v1.0.0 collapses that layer at the source — `template/<files>` instead of
`template/{{project_slug}}/<files>` — so freshly generated projects come
out flat. Existing projects on disk do not re-organise themselves; you
need to do it once, by hand. See [CHANGELOG entry for v1.0.0](../../CHANGELOG.md)
and the upstream fix it links to.

---

## What you'll see

**Before (v0.1.0-style emitted layout):**

```
my_project/                             ← repo root (git, .copier-answers.yml)
├── .copier-answers.yml
└── my_project/                         ← redundant inner directory
    ├── AGENTS.md
    ├── CLAUDE.md
    ├── CURSOR_RULES.md
    ├── README.md
    ├── pyproject.toml
    ├── .cursor/
    ├── .claude/
    ├── .github/
    ├── docs/
    ├── scripts/
    ├── src/
    │   └── my_project/                 ← Python package directory (correct)
    └── tests/
```

**After (v1.0.0 layout):**

```
my_project/                             ← repo root (git, .copier-answers.yml)
├── .copier-answers.yml
├── AGENTS.md
├── CLAUDE.md
├── CURSOR_RULES.md
├── README.md
├── pyproject.toml
├── .cursor/
├── .claude/
├── .github/
├── docs/
├── scripts/
├── src/
│   └── my_project/                     ← Python package directory (unchanged)
└── tests/
```

Note: only the **outer** `my_project/` layer disappears. The Python
package directory under `src/my_project/` stays exactly where it is.

## Prerequisites

- A clean git working tree on the affected project (`git status` shows nothing).
- A new branch dedicated to the migration so the diff is reviewable.
- PowerShell 7+ on Windows or bash on Linux/macOS.
- Replace `<project_slug>` in the commands below with your actual slug.

```powershell
# PowerShell
Set-Location <path-to-repo>
git checkout -b chore/migrate-to-projectsmith-v1.0.0
```

```bash
# Bash
cd <path-to-repo>
git checkout -b chore/migrate-to-projectsmith-v1.0.0
```

---

## PowerShell migration steps

```powershell
# Run from the repo root (the directory that contains the inner
# <project_slug>/ folder and .copier-answers.yml).
$ErrorActionPreference = "Stop"
$slug = "<project_slug>"   # e.g. "collaboration_protocol"

if (-not (Test-Path $slug -PathType Container)) {
    throw "Inner '$slug/' directory not found at $(Get-Location). Are you in the right repo?"
}

# 1. Move every tracked file inside <slug>/ up one level using git mv so
#    history is preserved.
$tracked = git ls-files "$slug/"
foreach ($src in $tracked) {
    $dst = $src.Substring("$slug/".Length)
    $dstDir = Split-Path -Parent $dst
    if ($dstDir -and -not (Test-Path $dstDir)) {
        New-Item -ItemType Directory -Path $dstDir -Force | Out-Null
    }
    git mv -- $src $dst
}

# 2. Remove the now-empty <slug>/ shell (also clears any untracked
#    artifacts such as __pycache__).
if (Test-Path $slug) {
    Remove-Item -Recurse -Force $slug
}

# 3. Stage and commit.
git add -A
git commit -m "chore: flatten projectsmith double-nesting (v0.1.0 -> v1.0.0)"
```

## Bash migration steps

```bash
# Run from the repo root (the directory that contains the inner
# <project_slug>/ folder and .copier-answers.yml).
set -euo pipefail
slug="<project_slug>"   # e.g. "collaboration_protocol"

if [ ! -d "$slug" ]; then
    echo "Inner '$slug/' directory not found in $(pwd). Are you in the right repo?" >&2
    exit 1
fi

# 1. Move every tracked file inside <slug>/ up one level using git mv so
#    history is preserved.
git ls-files "$slug/" | while IFS= read -r src; do
    dst="${src#$slug/}"
    dst_dir="$(dirname "$dst")"
    if [ "$dst_dir" != "." ] && [ ! -d "$dst_dir" ]; then
        mkdir -p "$dst_dir"
    fi
    git mv -- "$src" "$dst"
done

# 2. Remove the now-empty <slug>/ shell (also clears any untracked
#    artifacts such as __pycache__).
rm -rf "./$slug"

# 3. Stage and commit.
git add -A
git commit -m "chore: flatten projectsmith double-nesting (v0.1.0 -> v1.0.0)"
```

## After migration

Confirm the layout is now flat:

```powershell
Get-ChildItem -Force | Format-Table Mode, Name -AutoSize
```

You should see `AGENTS.md`, `CLAUDE.md`, `pyproject.toml`, `src/`, etc.
at the repo root — and crucially **no** `<project_slug>/` directory at
the top level.

History is preserved on each moved file:

```bash
git log --follow --oneline -- src/<project_slug>/config/env_spec.py | head
```

Run the standard verification:

```powershell
uv venv
uv lock
uv sync --dev
pwsh scripts/verify.ps1
```

Then update to v1.0.0 of the template:

```bash
copier update --vcs-ref=v1.0.0
```

This should run cleanly now that the on-disk layout matches the new
template. Review the resulting diff, accept scaffold-managed updates,
and re-run `verify`.

## Troubleshooting

- **`git mv` complains about destination already existing.** A previous
  flatten attempt may be partially applied. Inspect with `git status`,
  back out partial moves with `git restore` or `git mv` in reverse, then
  re-run from a clean state.
- **`__pycache__/` or other untracked files survive inside `<slug>/`.**
  `Remove-Item -Recurse -Force` and `rm -rf` (above) clear them. If you
  prefer to inspect first, run `Get-ChildItem -Recurse $slug -Force` /
  `find $slug -type f` before deleting.
- **`copier update` reports conflicts.** Expected for files you have
  customised locally (e.g., README.md, CLAUDE.md). Resolve as you would
  any normal merge.
